# 建立任務流程

當需要建立新任務時，依照此流程執行。

## 觸發時機

- 用戶明確要求建立任務
- 工作範圍較大，需要拆分追蹤

## 流程

### 1. 判斷任務類型

| 類型 | 特徵 | 使用模板 |
|------|------|---------|
| 簡單任務 | bug fix、小改動、單一檔案 | 簡單任務模板 |
| 設計任務 | 新功能、架構、多檔案、需要技術決策 | 設計任務模板 |

**判斷標準**：如果需要討論「怎麼做」而不只是「做什麼」，就是設計任務。

### 2. 確認任務資訊

向用戶確認：
- **任務名稱**：簡潔描述（用於檔名，kebab-case）
- **目標**：一句話說明要達成什麼
- **背景**：為什麼需要這個任務

若為設計任務，還需確認：
- **設計決策**：討論過的技術選擇和原因
- **檔案結構**：預計新增或修改哪些檔案
- **技術設計**：介面定義、資料結構

### 3. 建立任務文件

依 [template.md](template.md) 格式建立文件：

```
docs/dev/task/{task-name}.md
```

檔名規則：
- 使用 kebab-case
- 簡短但能辨識內容
- 例：`api-auth.md`、`refactor-db-layer.md`

### 4. 確保自足性

**關鍵檢查**：任務文件必須讓其他 session 能獨立執行。

檢查清單：
- [ ] 討論過的設計決策都已記錄
- [ ] 選擇 A 而非 B 的原因有說明
- [ ] 檔案結構完整列出
- [ ] 介面定義明確（不只是「實作 XXX」）
- [ ] 步驟具體可執行
- [ ] 驗證方式清楚

**錯誤示範**：
```markdown
## 步驟
1. 實作 LLM client
2. 加入 config 系統
```

**正確示範**：
```markdown
## 步驟
1. 建立 `llm/base.py`，定義 `Message` dataclass 和 `LLMClient` Protocol
2. 建立 `llm/providers/ollama.py`，實作 Ollama API 呼叫
3. 建立 `core/config.py`，載入 yaml 並解析路徑引用
```

### 5. 更新任務索引

在 [index.md](index.md) 的任務列表加入新任務：

```markdown
| [{task-name}.md]({task-name}.md) | {一句話說明} | 待處理 |
```

### 6. 回報用戶

告知用戶：
- 任務文件已建立
- 文件路徑
- 下一步動作（若有）

## 注意事項

- 任務應該是可完成的單位，避免過大
- 若任務太大，建議拆成多個子任務
- 建立後狀態預設為「待處理」
- **設計任務的文件長度通常較長是正常的**，重點是資訊完整
